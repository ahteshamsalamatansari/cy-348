System Overview

AI-powered carrier dispatch assistant for Relay On Demand. Carriers place orders, manage drivers, and view order status by chatting in natural language. The AI uses a guided multi-turn conversation flow to gather information before submitting orders, with a frontend confirmation gate that prevents accidental submissions. The system understands 4 form types and automatically selects the correct one based on the user's request.

1. Architecture & Service Flow

The system has a 3-tier architecture with a Cloudflare tunnel for public access:

User Browser | | HTTPS (Cloudflare Tunnel) v ai.appsscale.com ----cloudflared----> WSL (localhost) | | | /automationtest/* | | /api/automation/chat, /api/v2/* | v v chat_server.py (Port 9800) Model Coordinator (Port 9999) | Flask web server Enforces single-model loading | Serves HTML, proxies all calls Routes to LM Studio (Port 1234) | | /api/v2/* proxied to: v api_server.py (Port 9500) | Flask REST API server | Direct HTTP calls to Relay v relay_api.py (RelayAPI class) | HTTP client with auth v Relay REST API https://ar3lay0pndm4ndi.relayondemand.com/relayapi/api/

Request Flow: Chat Message

  1. User types message in automation_chat.html
  2. Frontend detects intent via regex patterns
  3. For LLM intents: POST /api/chat with messages array
  4. chat_server.py proxies to LM Studio API
  5. LM Studio streams response (SSE)
  6. Frontend parses for ORDER_DATA or API_ACTION commands

Request Flow: Order Placement (Multi-Turn with Confirmation Gate)

  1. User describes what they need (may be partial or complete info)
  2. LLM follows 6-step conversation flow: gathers missing info through follow-up questions
  3. LLM shows order summary and asks user to confirm
  4. User confirms → LLM outputs ORDER_DATA: {json}
  5. Frontend parses ORDER_DATA, validates (pickup address, classic driver ID)
  6. Confirmation Gate: Order stored in PENDING_ORDER, review card shown to user
  7. User types "confirm" / "submit" → Frontend submits; "cancel" → cleared; anything else → sent to LLM for modification
  8. POST /api/v2/place-order with order data
  9. chat_server.py proxies to api_server.py:9500
  10. api_server.py calls relay_api.build_order_data()
  11. relay_api.py calls POST PlaceOrder/NewOrder
  12. Result flows back through chain to browser
Safety: Orders are NEVER auto-submitted. The system has two confirmation layers: (1) the LLM asks for verbal confirmation before outputting ORDER_DATA, and (2) the frontend confirmation gate requires the user to explicitly type "confirm" before the API call is made.

System Prompt Structure

Built dynamically by buildSystemPrompt() function. The prompt enforces a strict multi-turn conversation flow to prevent the LLM from rushing to submit orders without gathering enough information.

6-Step Conversation Flow (CRITICAL)

The system prompt requires the LLM to follow these steps in order:

  1. UNDERSTAND — Read the user's message carefully, extract any provided info (addresses, times, pay, driver names, trailer types, etc.)
  2. IDENTIFY FORM TYPE — Determine which form fits: quick_entry (fast dispatch), smart_wizard (detailed requirements), or classic_entry (own fleet driver)
  3. CHECK WHAT IS MISSING — For the chosen form type, identify which required/important fields are still unknown
  4. ASK — Ask the user for any missing important information with specific questions
  5. SUMMARIZE & CONFIRM — Once enough info is collected, show a human-readable order summary and ask "Should I submit this order?" (Do NOT output ORDER_DATA yet)
  6. SUBMIT — ONLY after the user explicitly confirms (yes/confirm/submit/go), output ORDER_DATA: {json}

Key Prompt Behaviors

  • Info dump handling: If the user provides a lot of info in one message, the LLM extracts ALL relevant fields and only asks about what is still missing
  • Minimal info handling: If the user gives just a pickup address with no other details, the LLM asks about destination, timing, and pay preferences (does NOT submit with defaults)
  • Explicit defaults: Only uses default values when the user says "just use defaults", "that's fine", or "go ahead" after being asked
  • Error recovery: If an order fails, tells the user what went wrong and asks what they want to change (no auto-retry)
  • No accidental submissions: The prompt explicitly warns that ORDER_DATA should NEVER appear in examples or explanations, as the system executes ANY ORDER_DATA line as a real order

Prompt Sections (in order)

  1. Role definition — AI dispatch assistant for Relay On Demand
  2. 6-step conversation flow — UNDERSTAND → IDENTIFY → CHECK MISSING → ASK → SUMMARIZE → SUBMIT
  3. Important behaviors — Info dump handling, minimal info handling, defaults policy
  4. Output format rules — ORDER_DATA and API_ACTION command syntax
  5. 6 required fields — start_address, destination_address, when_want_driver, driver_pay_type, driver_book_duration, form_type
  6. Pay amount mapping — Which pay field to include based on driver_pay_type
  7. Default values — when_want_driver="2", driver_pay_type="0", flat_amount="0", driver_book_duration="2", form_type="quick_entry"
  8. Form type details & requirements — Including classic_entry's 3 required fields
  9. Optional fields — Trailer types, schedule, notes, task IDs
  10. API_ACTION commands — 12 actions the LLM can invoke for data queries
  11. Conversation examples — Multi-turn example (ask → answer → summary → confirm → submit) and info-dump example
  12. Dynamic context — Task types, trailer types, endorsement types, carrier name (injected from API_CONTEXT at runtime)

10. LLM Command Tags & Confirmation Gate

ORDER_DATA (with Confirmation Gate)

Output by LLM to place an order. Frontend parses with regex: /ORDER_DATA:?\s*(\{[^}]*\})/

ORDER_DATA: {"start_address": "123 Main St", "destination_address": "", "when_want_driver": "2", "driver_pay_type": "0", "driver_book_duration": "2", "form_type": "quick_entry", "flat_amount": "200"}

Important: ORDER_DATA is NOT auto-submitted. When the frontend detects ORDER_DATA in the LLM response, it:

  1. Parses the JSON and fills in missing defaults
  2. Validates required fields (pickup address, driver ID for classic entry)
  3. Stores the order in PENDING_ORDER state variable
  4. Displays an Order Review Card showing all fields in human-readable format
  5. Prompts user: "Type 'confirm' or 'submit' to place this order, or tell me what to change."

Order Review Card Fields

The review card displays these fields (when present):

  • Pickup address, Destination, When (ASAP/Scheduled/Timeframe)
  • Schedule date & time (if applicable)
  • Pay type & amount (e.g., "Flat $200")
  • Duration (hours), Form type, Trailer type, Driver name, Special notes

Confirmation Gate Responses

User ResponseAction
Confirm words: yes, confirm, submit, go, do it, place it, go ahead, send it, ok, okay, sure, proceedSubmits order via POST /api/v2/place-order
Cancel words: no, cancel, nevermind, stop, don't, nah, nopeClears pending order, shows cancellation message
Anything else (e.g., "change the pay to $300")Injects pending order context into chat history, sends to LLM for modification. LLM shows updated summary.

11b. Multi-Turn Conversation & Confirmation Gate

Design Philosophy

The system uses a two-layer safety model to prevent accidental order submissions:

  1. LLM Layer: The system prompt enforces a 6-step conversation flow. The LLM must ask questions, gather info, show a summary, and get verbal confirmation BEFORE outputting ORDER_DATA.
  2. Frontend Layer: Even after the LLM outputs ORDER_DATA, the frontend does NOT auto-submit. It stores the order in PENDING_ORDER and shows a review card requiring explicit user confirmation.

Conversation Flow Diagram

User: "I need a driver at 123 Main St" | v LLM Step 1-3: Extract info, identify form type, check missing fields | v LLM Step 4: "Got it! Pickup at 123 Main St. A few questions: 1. Destination? 2. When? 3. Pay preferences?" | v User: "No destination, ASAP, flat rate $150" | v LLM Step 5: "Order summary: - Pickup: 123 Main St - When: ASAP - Pay: Flat $150 Should I submit this?" | v User: "Yes" | v LLM Step 6: Outputs ORDER_DATA: {json} | v Frontend: Parses JSON, validates, stores in PENDING_ORDER | v Frontend: Shows ORDER REVIEW card with all fields "Type 'confirm' to place this order, or tell me what to change." | v User: "confirm" ---> POST /api/v2/place-order ---> Success/Failure User: "cancel" ---> Clear PENDING_ORDER, show cancellation User: "change X" ---> Send to LLM with order context for modification

PENDING_ORDER Lifecycle

EventPENDING_ORDER StateWhat Happens
LLM outputs ORDER_DATASet to parsed order objectReview card shown, awaiting confirmation
User says "confirm"/"submit"Cleared to nullOrder submitted to API, result shown
User says "cancel"/"no"Cleared to nullCancellation message shown
User says anything elseCleared to nullOrder context injected into chat, LLM handles modification and shows updated summary
Order submission failsCleared to nullError message shown, user asked what to change

Order Modification Flow

When the user responds with a modification request (e.g., "change the pay to $300") while PENDING_ORDER is active:

  1. Previous order data is serialized to JSON and injected into CHAT_HISTORY as a system message
  2. PENDING_ORDER is cleared
  3. Message falls through to normal LLM handling
  4. LLM receives both the user's change request and the previous order context
  5. LLM applies changes and shows an updated summary (without outputting ORDER_DATA)
  6. User confirms again → LLM outputs new ORDER_DATA → new review card shown

When ORDER_DATA is parsed, the frontend fills in any missing required fields with defaults:
