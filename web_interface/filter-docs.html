System Overview

The Smart Filter is an AI-powered content analysis tool that detects obfuscated contact information (phone numbers, emails, social media handles) and circumvention attempts in user messages. The server automatically tracks the last 5 messages per user_id to detect information split across multiple messages — no client-side history management needed.

1. Architecture
Client App / Browser | | POST /api/filter { message, user_id } v chat_server.py (Port 9800) | Looks up user_id → retrieves last 5 messages | Builds system prompt + user history | Calls LM Studio API v LM Studio (localhost:1234) | unsloth/qwen3.5-35b-a3b model | Analyzes NEW message + conversation history v JSON Response: {safe, category, reason, user_id, context_size} | v Server saves message to user's history

3. API Endpoint

POST /api/filter

Request Body

{ "message": "The message text to analyze", // required "user_id": "unique-user-identifier", // recommended "username": "display-name", // optional alias for user_id "context": ["manual", "override", "messages"] // optional (overrides auto-history) }

Request Fields

FieldTypeRequiredDescription
messagestringYesThe message text to analyze
user_idstringRecommendedUnique user identifier. Enables per-user message history tracking on the server. If omitted, defaults to "anonymous" (shared history).
usernamestringNoAlias for user_id. If user_id is not provided, username is used instead.
contextstring[]NoManual context override. If provided, these messages are used instead of the server-side auto-tracked history. Use this if your app manages its own message history.

Response

{ "safe": true | false, "category": "CLEAN" | "PHONE" | "EMAIL" | "SOCIAL" | "CIRCUMVENTION", "reason": "Explanation of why this was flagged or marked clean", "user_id": "the-user-id-used", "context_size": 3 }

Response Fields

FieldTypeDescription
safebooleantrue if no contact info detected, false if flagged
categorystringDetection category (see below)
reasonstringHuman-readable explanation
user_idstringThe user_id that was used for this analysis
context_sizeintegerNumber of previous messages included as context (0-5)

Per-User History Tracking

Server-side history: The server automatically tracks the last 5 messages per user_id. No client-side history management needed — just send user_id + message with each call.

How it works:
• Each message is saved to the user's history AFTER analysis
• The next call with the same user_id includes previous messages as context
• Sessions auto-expire after 30 minutes of inactivity
• Different user_id values have completely isolated histories
• Maximum 5 messages retained per user (sliding window)

CIRCUMVENTIONAttempt to move conversation off-platform"let's talk privately", "contact me outside this app"
Split Detection: The server tracks per-user message history to detect information split across multiple messages. For example:
• Message 1 (user_id: "driver-123"): "My number starts with five five five" → PHONE
• Message 2 (user_id: "driver-123"): "and ends with one two three four" → PHONE (split detected: 555 + 1234)
• Message 2 (user_id: "driver-456"): "and ends with one two three four" → CLEAN (different user, no context)

No client-side history needed — the server handles it automatically via user_id.
5. Context Window System

The filter maintains a 5-message sliding window (configurable via CONTEXT_WINDOW_SIZE). Each new message is analyzed with up to 5 previous messages as context, allowing the AI to detect split information.

How It Works

  1. Frontend stores all messages in chatHistory[] array
  2. On each analysis, the last 5 messages are sent as context
  3. The LLM system prompt instructs: "Consider ALL messages together. A phone number may be split across lines."
  4. The LLM returns its analysis considering the full context

Server-Side System Prompt (in chat_server.py)

The system prompt instructs the LLM to:

  • Detect phone numbers, emails, social media handles, circumvention attempts
  • Consider messages split across multiple lines
  • Detect obfuscation (spelled out numbers, symbols replacing characters)
  • Return JSON with exactly 3 fields: safe, category, reason
  • Categories: CLEAN, PHONE, EMAIL, SOCIAL, CIRCUMVENTION

7. Integration Examples

cURL

curl -X POST https://ai.appsscale.com/api/filter \ -H "Content-Type: application/json" \ -d '{"message": "Call me at five oh five 867 5309"}'

Python

import requests response = requests.post( "https://ai.appsscale.com/api/filter", json={ "message": "My number is 5-0-5 then 867-5309", "context": ["I want to discuss this privately"] } ) result = response.json() print(f"Safe: {result['safe']}, Category: {result['category']}") print(f"Reason: {result['reason']}")

JavaScript (fetch)

const response = await fetch('https://ai.appsscale.com/api/filter', { method: 'POST', headers: { 'Content-Type': 'application/json' }, body: JSON.stringify({ message: 'Email me at john [at] gmail [dot] com', context: [] }) }); const result = await response.json(); console.log(`Safe: ${result.safe}, Category: ${result.category}`);

Batch Analysis (Python)

import requests messages = [ "The box weighs 50kg", "Call me at five oh five", "My snap handle is @john123", ] for msg in messages: r = requests.post( "https://ai.appsscale.com/api/filter", json={"message": msg} ).json() status = "SAFE" if r["safe"] else "FLAGGED" print(f"[{status}] {r['category']}: {msg[:50]}")

10. Troubleshooting

Filter Not Responding

# Check carrier-chat service systemctl status carrier-chat # Test filter endpoint locally curl -X POST http://localhost:9800/api/filter \ -H "Content-Type: application/json" \ -d '{"message": "test"}'

False Positives/Negatives

The LLM model determines detection accuracy. To improve:

  • Update system prompt in chat_server.py (filter_message function)
  • Add examples to the prompt
  • Increase context window size in filter.html
  • Try different model (change MODEL_NAME)
