System Overview

This is an AI-powered carrier dispatch assistant for Relay On Demand. It lets carriers place orders, manage drivers, and view order status by chatting in natural language. The AI model understands 4 form types and automatically selects the right one based on what the user says.


What It Does
  • Place dispatch orders via natural language chat (no forms to fill)
  • Automatically select the correct form type (Quick Entry, Smart Wizard, Classic Entry, Multi-Driver Blocks)
  • Query orders, drivers, dashboard data via conversational commands
  • Validate inputs and ask for missing info before submitting
  • Stream AI responses token-by-token for fast feedback

LM Studio API (AI Engine)

The AI engine runs on LM Studio on a local machine, exposed via Cloudflare tunnel as api.appsscale.com.


Configuration
Endpoint: https://api.appsscale.com/v1/chat/completions Local: http://localhost:1234/v1/chat/completions Model: unsloth/qwen3.5-35b-a3b Format: OpenAI-compatible API (same request/response format)

How It's Used

The chat_server.py (port 9800) acts as a proxy. When the frontend calls POST /api/automation/chat, the server injects the MODEL_NAME into the payload and forwards it to the LM Studio API. This hides the model name from the browser client.


Streaming

When stream: true is set in the request, the server uses Server-Sent Events (SSE) to stream tokens back. Each chunk is a data: line with a JSON object containing choices[0].delta.content. The frontend reads these via ReadableStream and appends text to the message bubble in real-time.


Request Format
{ "messages": [ {"role": "system", "content": "You are a carrier dispatch assistant..."}, {"role": "user", "content": "Send a driver to 123 Main St"} ], "temperature": 0.3, "max_tokens": 16000, "stream": true }

Think Tags

The Qwen 3 model outputs <think>...</think> reasoning blocks. The frontend detects these during streaming and shows "Thinking..." until the </think> tag appears, then displays only the text after it.

API Server (api_server.py — Port 9500)

Flask API server providing two sets of endpoints:

  • /api/* (v1) — Playwright browser automation (legacy, slower but handles any page)
  • /api/v2/* — Direct HTTP API calls via relay_api.py (fast, reliable, current)

Relay API Authentication

The server maintains a single shared RelayAPI instance (_relay_api) that auto-authenticates on first request. Authentication uses CARRIER_USERNAME / CARRIER_PASSWORD environment variables (defaults to test account). A thread lock (_relay_api_lock) ensures thread-safe access.


v2 Order Placement (POST /api/v2/place-order)

Accepts either simplified field names (lowercase, snake_case from the LLM) or raw Relay API field names (PascalCase). If start_address is present, it calls relay_api.build_order_data() to map simplified names to the full Relay API payload (~55 fields). The form_type field controls which flags get set:

form_type="quick_entry" → IsOrderCreatedFromNewFlow=true, FromSmartWizard=false form_type="smart_wizard" → IsOrderCreatedFromNewFlow=true, FromSmartWizard=true form_type="classic_entry" → CarrierDriverID and CarrierDriverName required

Simplified → Relay API Field Mapping
start_address → StartAddress destination_address → DestinationAddress when_want_driver → WhenYouWantDriver ("0"/"1"/"2") driver_pay_type → DriverPayType ("0"-"4") driver_book_duration→ DriverBookDurations flat_amount → FlatAmount hourly_amount → HourlyFlatAmount per_load_amount → PerLoadAmount per_mile_amount → PerMileAmount per_week_amount → PerWeekAmount trailer_type_id → TrailerTypeID truck_unit_number → TruckUnitNumber trailer_number → TrailerNumber special_notes → SpecialNotes internal_notes → InternalNotes schedule_time → ScheduleTime carrier_driver_id → CarrierDriverID carrier_driver_name → CarrierDriverName schedule_date → ScheduleDate start_task_id → StartAddressTaskID end_task_id → EndAddressTaskID
Relay API Client (relay_api.py)

Python class RelayAPI that makes direct HTTP calls to the Relay On Demand REST API. This was reverse-engineered from the Angular JS bundles of the carrier website.


Discovery

The Relay REST API was discovered by analyzing two JavaScript bundle files from https://carrier.relayondemand.com/Carrier/:

main.95acff2d97c0e868eca2.bundle.js (1.0 MB) - Core app, auth, services 0.267e3fe7d8f30548a65f.chunk.js (2.4 MB) - Lazy-loaded ClientModule

This revealed 144 REST API endpoints, the full PlaceOrder/NewOrder data model (~55 fields), enum values, and SignalR hub definitions. All form types (Quick Entry, Smart Wizard, Classic Entry) submit to the same single endpoint: POST PlaceOrder/NewOrder.

Streaming Implementation

callLLMStream() uses fetch() with a ReadableStream reader. It reads SSE lines (data: {...}), parses JSON, extracts choices[0].delta.content, appends to fullText, and updates the message bubble. Think tags are handled: while inside <think> block, display shows "Thinking...". After </think>, only post-think text is shown.

Intent Detection System

The detectIntent(message) function classifies user messages using regex patterns. Direct data intents bypass the LLM entirely for faster response.


Intent Priority Order (top wins)
1. form_info → /form.?type|what.*form|explain.*form|options/ Action: Show static form type descriptions (no LLM) 2. place_order → /dispatch|send.*driver|place.*order|create.*order|new.*order/ Action: Send to LLM with system prompt 3. smart_wizard → /wizard|step.by.step|guide|help.*plac/ Action: Send to LLM (model outputs ORDER_DATA with smart_wizard) 4. multi_driver_blocks → /block|multi.*driver|recurring|weekly.*block/ Action: Send to LLM (model explains MDB) 5. classic_entry → /classic|assign.*my.*driver|own.*driver|fleet/ Action: Send to LLM (model asks for driver ID) 6. view_orders → /(current|my|active|show|list).*orders?/ Action: GET /api/v2/orders (no LLM) 7. cancel_order → /cancel.*(order|dispatch)/ Action: Send to LLM 8. order_detail → /order.*(detail|#|\d{3,})/ Action: Send to LLM 9. drivers → /(show|list|available|preferred|my).*drivers?/ Action: GET /api/v2/drivers (no LLM) 10. form_info (2) → /how.*(order|dispatch|place)/ Action: Show static form type descriptions 11. legacy → /analyze|screenshot|page|navigate/ Action: Send to LLM (browser automation) 12. chat (default) → Everything else Action: Send to LLM
Why form_info is checked BEFORE place_order: "What form types can I use to place an order?" would otherwise match "place.*order" and go to the LLM instead of showing the static form info.
LLM System Prompt & Command Tags

The buildSystemPrompt() function constructs a detailed system prompt that tells the LLM how to respond. It includes output format rules, required fields, form type selection logic, examples, and dynamically-loaded reference data.


Output Commands

The LLM communicates actions via two special tags that the frontend regex-parses:

ORDER_DATA: {"start_address":"...","form_type":"quick_entry",...} → Frontend parses JSON, validates, calls POST /api/v2/place-order API_ACTION: {"action":"view_orders"} → Frontend calls executeAPIAction(), displays formatted results

Rules Enforced by Prompt
  • ORDER_DATA must be on a SINGLE line (no line breaks in JSON)
  • All JSON values must be strings (quoted)
  • No markdown code blocks around JSON
  • All 6 required fields must be present
  • Never submit ORDER_DATA without start_address
  • Never submit classic_entry without carrier_driver_id
  • If user just says "place an order" with no details, ask for pickup address

6 Required Fields
1. start_address Pickup address (must not be empty) 2. destination_address Drop-off address ("" if not specified) 3. when_want_driver "0"=exact date/time, "1"=timeframe, "2"=not sure 4. driver_pay_type "0"=flat, "1"=hourly, "2"=weekly, "3"=per load, "4"=per mile 5. driver_book_duration Hours as string (e.g. "2") 6. form_type "quick_entry" | "smart_wizard" | "classic_entry"

Dynamic Context Injection

If API_CONTEXT is loaded (from /api/v2/context), the prompt appends:

  • Task Types — List of DefaultParameterID + name + hours (for start_task_id / end_task_id)
  • Trailer Types — List of TrailerTypeID + name
  • Endorsement Types — List of EndorsementID + type
  • Carrier Name — Current carrier identity

Available API_ACTION Actions
view_orders → GET /api/v2/orders?type=now view_scheduled_orders → GET /api/v2/orders?type=scheduled view_all_orders → GET /api/v2/orders?type=all view_order_detail → GET /api/v2/orders/{id} (params.order_id) cancel_order → POST /api/v2/orders/{id}/cancel (params.order_id, params.reason) view_drivers → GET /api/v2/drivers?type=own view_preferred_drivers → GET /api/v2/drivers?type=preferred view_active_drivers → GET /api/v2/drivers?type=active view_quick_orders → GET /api/v2/quick-orders view_multi_driver_blocks → GET /api/v2/multi-driver-blocks estimate → POST /api/v2/estimate (params.start_address, params.destination_address) view_dashboard → GET /api/v2/dashboard
Form Types In Detail
1. Quick Entry (Default)
Trigger words: "send a driver", "dispatch", "quick", or just an address API flags: IsOrderCreatedFromNewFlow=true, FromSmartWizard=false form_type: "quick_entry" Fields: start_address, destination_address, when_want_driver, driver_pay_type, pay amount, driver_book_duration Use case: Simple dispatches with minimal detail

2. Smart Wizard
Trigger words: "help me", "guide me", "wizard", "step by step", or detailed info (destination + trailer + schedule + pay) API flags: IsOrderCreatedFromNewFlow=true, FromSmartWizard=true form_type: "smart_wizard" Fields: All Quick Entry fields PLUS trailer_type_id, schedule_time, endorsement types, additional stops Use case: Complex orders, per-mile/per-load pricing, reefer/flatbed Note: If user says "guide me" but provides ALL details in one message, the LLM outputs ORDER_DATA immediately (no follow-up questions)

3. Classic Entry
Trigger words: "my driver", "assign to [name]", "our driver", "fleet" API flags: CarrierDriverID and CarrierDriverName required form_type: "classic_entry" Extra fields: carrier_driver_id, carrier_driver_name, schedule_date Use case: Assign order to your own fleet driver (not a Relay network driver) Validation: Frontend blocks submission if carrier_driver_id is empty, fetches driver list and asks user to pick one

4. Multi-Driver Blocks
Trigger words: "block", "multi driver", "recurring", "weekly block" API endpoint: POST PlaceOrder/NewRelayFromMultiDriverBlock/{blockId} No form_type: (not submitted via ORDER_DATA) Use case: Create orders from pre-booked recurring driver blocks Flow: LLM explains MDB → shows existing blocks via API_ACTION → user selects block → LLM creates relay from block
Frontend Validation & Defaults

Three layers of validation ensure no bad data reaches the Relay API:


Layer 1: LLM Prompt Rules

The system prompt instructs the model to never output ORDER_DATA without a start_address, and to ask for missing info. This works ~90% of the time.


Layer 2: Frontend JavaScript Defaults

After parsing ORDER_DATA, the frontend fills missing required fields with defaults:

when_want_driver: "0" if schedule_date/time present, else "2" driver_pay_type: "0" (flat rate) driver_book_duration:"2" (2 hours) form_type: "quick_entry" destination_address: "" (empty string) Pay amount auto-fill: driver_pay_type "0" → flat_amount (default "0") driver_pay_type "1" → hourly_amount (default "0") driver_pay_type "2" → per_week_amount (default "0") driver_pay_type "3" → per_load_amount (default "0") driver_pay_type "4" → per_mile_amount (default "0")

Layer 3: Frontend Validation Checks
Classic Entry without driver ID: → Fetches driver list from /api/v2/drivers?type=own → Displays numbered list: "1. John Smith (ID: 3408)" → Asks user to pick a driver → Does NOT submit ORDER_DATA Missing pickup address (empty start_address): → Shows: "I need a pickup address to place the order." → Does NOT submit ORDER_DATA
Smart Filter (filter.html)

A separate tool at /filter.html that uses the same Qwen 3 model to detect contact information or platform circumvention in messages.


Access
UI: https://ai.appsscale.com/filter.html API: POST https://ai.appsscale.com/api/filter Docs: GET https://ai.appsscale.com/api/filter/docs

API Request
POST /api/filter { "message": "Call me at five oh five 867 5309", "context": ["previous message 1", "previous message 2"] // optional }

API Response
{ "safe": false, "category": "PHONE", // CLEAN | PHONE | EMAIL | SOCIAL | CIRCUMVENTION "reason": "Phone number detected: 505-867-5309" }

The filter uses the same MODEL_NAME and LM_STUDIO_API as the chat. It sends a specialized system prompt that instructs the model to analyze messages for contact info. Context-aware: can detect numbers split across multiple messages (e.g., "555" then "1234").
